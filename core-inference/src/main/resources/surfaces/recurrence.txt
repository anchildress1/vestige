## Surface: Recurrence

Match the entry against retrieved history chunks (provided in the user content block).

What to capture:

- Whether this entry repeats a behavior, vocabulary pattern, or state shift seen in a retrieved historical entry.
- Which retrieved chunk(s) are the closest match, by `pattern_id` if present in the chunk metadata.
- Whether the recurrence is exact (same activity + same context) or partial (same activity, different context).

What goes in the schema:

- Populate `recurrence_link` with the `pattern_id` of the matched historical pattern, or null when no match.
- Populate `recurrence_kind` with `exact` or `partial` when `recurrence_link` is non-null.
- Do not encode the pattern id into `tags`. The structured `recurrence_link` field is the only place pattern ids belong; tags must remain short kebab-case tokens per the output schema.

Constraints:

- Do not invent a `pattern_id`. Use only IDs present in the retrieved-history block.
- Retrieved chunks without a `pattern_id` are context-only. They can inform the reading, but they cannot populate `recurrence_link`.
- Do not infer recurrence from semantic similarity if no retrieved chunk references it. The retrieval layer is the only source of historical context for this surface.
- If the user content contains no retrieved-history block, `recurrence_link` is null. This is normal for early entries.
