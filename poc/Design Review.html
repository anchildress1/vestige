Vestige

a structural read

The prototype is atmospherically strong and the information design holds up. Before it goes to the builder, the seams need attention: two visual systems share one app, one data model has been forked four ways, and a handful of routes do things they shouldn't. None of it is fatal. All of it is cheaper to fix now than after the build.

reviewer · principal designer
scope · 11 source files · 12 screens
— hand off when green

What to fix before build.

Five structural decisions account for most of the noise in this prototype. Resolve these and the rest of the punch list collapses to polish.

  1. Collapse the legacy palette. Tokens still ship pink/blue/live aliases mapped to the new moonstone family. Half the screens use the old names. Pick one set, delete the other.
  2. Unify the surface system. Capture / Patterns / Reading use glass + noise + large radii. Onboarding / Status / Roast / Error use flat cards + 6px corners. They look like two apps. Pick one card primitive and use it everywhere.
  3. Make Roast a real overlay. Today it's rendered by stacking PatternsScreen and RoastScreen in a fragment so the sheet has a backdrop. Build a proper sheet primitive with its own route or modal stack.
  4. One persona record, not four. PERSONAS, VOICES, ROAST, INLINE_REPLIES are four maps keyed by the same id. Fold them into one shape so a new persona is one object, not four edits.
  5. Co-locate pattern data with patterns. The 30-day hit map is duplicated in three places (PatternsScreen, PatternDetailScreen, Capture teaser). Move traceHits onto each pattern record.
Critical
Real bug, data duplication, or structural mismatch the builder will inherit. Fix before implementation.
Structural
Architecture / boundary issue. Doesn't break the prototype but will compound during build.
Polish
Inconsistency, dead code, or small craft issue. Cheap to fix; ignore at your discretion.
01

Architecture & routing.

5 findings
A.01
Critical app.jsx · screens-roast.jsx

Roast is rendered by stacking two screens in a fragment.

The 'roast' route returns <PatternsScreen /> followed by <RoastScreen /> inside a fragment. RoastScreen then positions itself absolute over its sibling so the dim Patterns view shows through. There is no real modal/overlay primitive — Roast is a screen pretending to be a sheet. The dev-nav button "10 Roast" only works because of this trick.

Where
app.jsx · case 'roast' in renderScreen()
screens-roast.jsx · RoastScreen uses position: absolute on its root
Fix
Lift modal/sheet state out of the screen switch. A <Sheet open onDismiss> primitive that renders over the current route is the cleanest path. Roast becomes a sheet opened from Patterns, not a peer route. Side benefit: removes the screen-stacking hack and makes Pattern Detail's "Resolved" / "Snooze" actions reusable as sheets later.
A.02
Critical screens-reading.jsx · tokens.jsx

ReadingScreen passes left= to AppShellTop, which doesn't accept it.

AppShellTop accepts title, modelState, onStatusTap, right, dense. ReadingScreen passes a left prop holding a back button. It's silently dropped — there's no back affordance on the Reading screen except the model-status pill, which opens the status modal instead. Real bug, easy to miss in dev-nav demos because users land on Reading via a button rather than a flow.

Where
screens-reading.jsx · <AppShellTop left={…}>
tokens.jsx · function AppShellTop({ title, modelState, onStatusTap, right, dense })
Fix
Either teach AppShellTop a left slot (recommended — Pattern Detail and Persona screens currently put their back button on the right, which is wrong for Android), or remove the left prop from Reading and pick the right pattern globally. Pick a side and apply it to every detail screen.
A.03
Structural screens-onboarding.jsx · tokens.jsx

Status pill is tappable during onboarding — and traps the user.

AppShellTop is mounted in onboarding with no onStatusTap override, so tapping the model dot dispatches vestige:open-status. That handler runs setScreen('status') globally — which yanks the user out of onboarding into the Local Model screen, with no path back into the flow. First-run trap.

Where
app.jsx · global 'vestige:open-status' listener
screens-onboarding.jsx · <AppShellTop modelState=…> with no override
Fix
Make the status pill a no-op (or non-interactive) inside onboarding. Better: move the global event listener into a context that's only active once onboarding is done.
A.04
Structural screens-reading.jsx · screens-patterns.jsx

"Reading" has two contradictory entry points.

ReadingScreen is reached two ways: (a) from the dev nav as its own route, and (b) by tapping a source row in Pattern Detail, which calls onOpenReading(source) and passes the source object. ReadingScreen ignores the argument and always renders SAMPLE_READING. So tapping any source row on any pattern shows the same Crashed-at-3pm reading. Boundary between "the reading for THIS entry" and "the reading screen" is unclear.

Where
screens-patterns.jsx · onClick={() => onOpenReading && onOpenReading(s)}
screens-reading.jsx · const r = SAMPLE_READING; — argument never read
Fix
Either commit to ReadingScreen receiving an entry id (and put a real readings map in data.jsx), or remove the source-row affordance until that data exists. Don't ship a tap target that lies.
A.05
Polish app.jsx

Destructive confirm dumps users into Onboarding.

On confirm of "Wipe everything", the app routes to 'onboarding'. The intent is clear (wiped → first-run), but the transition is jarring: no acknowledgment beat, no "wiped" success state, just an instant jump. For a destructive action this is too quiet.

Fix
Add a brief "Wiped." beat (one screen, mono, 1.5s, single line) before onboarding loads. Matches the app's restraint and confirms the action took effect.
02

Design system & tokens.

4 findings
D.01
Critical tokens.jsx

Two visual systems live in the app at once.

System A — atmospheric: Capture, Patterns, Pattern Detail, Reading. Translucent surfaces (oklch(22% 0.022 252 / 0.7)), backdropFilter: blur, inline noise overlays, large radii (rL: 20, rXL: 28), moonstone halos.
System B — flat: Onboarding, Local Model, Persona Selector, Roast, Error, Destructive. Solid V.s1 cards, hard borderRadius: 6 magic numbers, no glass, no noise, legacy pink/blue palette.
The same primitive — "a card" — has two completely different implementations depending on which file you're in.

Worst offenders
PatternCard (atmospheric, rL=20) vs PersonaCard (flat, 6) — both are "selectable list cards"
FieldCard (Reading) vs the row stack in ModelStatusScreen — both are "key-value blocks"
Fix
Pick one. The atmospheric system is the brand — commit to it. Build three shared primitives: <Surface> (the glass + noise card), <Row> (a key/value line), <ListCard> (selectable variant of Surface). Migrate Onboarding, Status, Persona, Roast, Error, Destructive onto them. The "flat" treatment in onboarding currently reads as unfinished, not as a deliberate counterpoint.
D.02
Critical tokens.jsx · all screens

Legacy color aliases shipped, and half the app still references them.

Tokens carry pink / pinkDim / pinkRule / blue / blueDim / live / liveDim with a comment marking them as legacy aliases mapped onto the new glow / vapor / pulse palette. Onboarding, Persona, Roast, Error, Destructive, ModelStatus all reference the legacy names. Capture, Patterns, Reading reference the new ones. Visually they currently render the same color — but the moment someone retunes the moonstone hue, only half the app moves.

Fix
Codemod the screens onto the canonical names (V.glow, V.vapor, V.pulse), then delete the alias block. Land this before the builder picks it up; cleaning it after means searching across both vocabularies.
D.03
Structural all screens

Magic radii in production code.

The radii scale (rPill, rXL: 28, rL: 20, rM: 14, rS: 8) is intentional and well-thought. But raw borderRadius: 6 appears in roughly a dozen places — PersonaCard, model download bar, error/destructive containers, Roast wipe button, etc. Six is below rS; it was never declared.

Fix
Either add an rXS: 6 token if 6px is a real step, or push offenders to rS: 8. No ambient magic numbers.
D.04
Polish tokens.jsx

Per-component <style> tags duplicate keyframes.

vesPulse, vesIn, vesShimmer, vesBreath, vesSpin, vesDrift1/2, vesFade, vesSlide are each declared in whichever component first uses them. vesSpin is declared twice. They'll drift.

Fix
Move all keyframes to a single global <style> block injected from tokens.jsx. Components reference by name only.
03

Data model.

3 findings
M.01
Critical data.jsx

Persona content is forked across four maps.

PERSONAS holds id + name + tagline. VOICES holds the example transcript. ROAST holds the roast lines. INLINE_REPLIES holds mid-session replies. Four separate objects keyed by the same persona id. Adding a fourth persona ("Therapist", say) is four edits, three of which are easy to forget. INLINE_REPLIES is currently declared but isn't actually consumed anywhere.

Fix
Collapse to one shape: { id, name, tagline, transcript, roast, inlineReplies }. Each persona is one record. Drop INLINE_REPLIES if the recording flow doesn't actually need the third surface — the prototype scripts pre-rendered turns from VOICES.transcript instead, which suggests it was abandoned.
M.02
Critical screens-patterns.jsx · screens-capture.jsx

The 30-day hit map is duplicated three times.

A literal hitMap object lives in PatternsScreen and again in PatternDetailScreen. The Capture teaser uses a different hardcoded spread ([2, 9, 16, 23, 5, 12, 19, 26, 28]) that doesn't correspond to any single pattern. Same data, three places, one of them lying.

Fix
Move traceHits: number[] onto each PATTERN record. The Capture teaser should display either an aggregate (union of all active patterns' hits) or one representative pattern's hits — pick on intent and make it derive from the source.
M.03
Structural screens-patterns.jsx

Pattern lifecycle (active / snoozed / resolved) is fictitious.

Patterns expose three states in the list (Active / Snoozed / Resolved) — but the partition is hardcoded with dormant.slice(0, 1). PatternDetailScreen's three lifecycle buttons (Snooze / Resolved / Dismiss) all call onClose. There is no transition path. Builder should not assume this is wired.

Fix
Add status: 'active' | 'snoozed' | 'resolved' to each pattern record; wire the detail buttons to mutate it; partition the list by status. Document this as "the prototype's biggest fiction" so the builder knows where to start.
04

Component boundaries.

4 findings
C.01
Structural screens-capture.jsx

MistHero overlays state on top of itself instead of accepting it.

MistHero renders its own 14px ink dot at the center. RecordingScreen then renders an absolutely-positioned overlay (square box, spinner, etc.) on top of the dot to express listening / thinking states. So the stone has two stacked center marks; the inner dot is hidden by the overlay rather than swapped. Re-skinning the state expression means editing two files.

Fix
Hoist the center mark to a state prop on MistHero: <MistHero state="idle | listening | thinking | replied" />. One component owns its full rendering surface.
C.02
Structural screens-capture.jsx · tokens.jsx

IconSlot is being used as production chrome.

IconSlot is documented as a palimpsest placeholder — a watermark for an icon that hasn't been drawn. The persona pill on Capture uses it with label="·" as a real visual element. Reads as unfinished, not deliberate.

Fix
Replace with a small persona dot (the moonstone glow at 12–14px) or a 2-letter initial chip in mono. Reserve IconSlot for actual placeholders during build-out.
C.03
Polish tokens.jsx

AppShellTop has dead props.

title is consumed only by ModelStatusScreen, which passes it as the empty string. dense is declared but never set anywhere. The centered-title slot is structurally there but visually never used.

Fix
Drop title and dense. If a future detail screen needs a centered title, add it back deliberately.
C.04
Polish app.jsx · all screens

Aliased React hook destructures.

Each file destructures useState and useEffect under unique aliases — useSA, useSC, useSO, useSP, useSR (twice, in roast and reading). Babel's per-script scoping makes this safe today, but the aliases are pure noise and confuse new readers.

Fix
Standardize on const { useState, useEffect } = React; at the top of every file. The alias dance is a vestige of an earlier worry about scope collisions that doesn't apply.
05

Tweaks & dev surface.

2 findings
T.01
Polish app.jsx

Dead tweak: density.

TWEAK_DEFAULTS.density = 'regular' is declared and persisted but never read or exposed in the panel. Either wire it (Compact / Regular density toggle on Patterns + Reading) or remove it.

T.02
Polish screens-capture.jsx · app.jsx

"History" footer button opens the Error screen.

The Capture footer's history link is wired to onError. It's a dev-nav shortcut that ships looking like a real entry-point. A user testing the build will tap this and get a network error.

Fix
Either remove the link until History exists, or hide it behind the Tweaks toggle so it only appears in dev mode.
06

Closeout checklist.

for the builder

Collapse persona content into one record per persona. M.01 — easiest big win
Move traceHits onto pattern records; remove all hitMaps. M.02
